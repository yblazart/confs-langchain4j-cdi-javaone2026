The CDI solution


@RegisterAIService(chatModelName = "my-model",
                   tools = BookingTools.class)
public interface Assistant {

    @SystemMessage("You are a booking assistant")
    String chat(@UserMessage String message);
}
    

// And in your business code:
@Inject
Assistant assistant;

// That's it. Really.
String response = assistant.chat("Book me a table");
And here is what it looks like with langchain4j-cdi. One interface, one annotation, and an @Inject. That's it. No builder, no manual instantiation. The CDI container does all the work. For those who know CDI, it's exactly the same pattern as a JPA repository or a REST client. Interface + annotation = injectable bean.

Yes, it also runs on Quarkus. They can retire their extension. 😏

Emmanuel explains the duality of CDI extensions here. CDI 4.1 introduced Build Compatible Extensions -- a simpler and more performant API for creating extensions. Quarkus and Helidon use them for compile-time processing. Traditional servers like WildFly use Portable Extensions, which do the same work at runtime. langchain4j-cdi provides both implementations. It's completely transparent: you don't change anything in your code, the right module activates based on your server. And yes, it also runs on Quarkus! Under the hood 🔧

1. The extension scans for @RegisterAIService

2. For each interface found, it reads the attributes:
chatModelName, tools, chatMemoryProviderName, retrievalAugmentorName

3. It creates a synthetic CDI bean:
→ LangChain4j proxy wrapped in a managed CDI bean

4. Components are resolved via an SPI (LLMConfig):
dev.langchain4j.cdi.plugin.<name>.class=...
→ MicroProfile Config implementation provided by langchain4j-cdi-config → WildFly implementation provided by wildfly-ai-feature-pack

5. @Inject Assistant assistant; → ✨ It works!

Let's detail the mechanism. At startup (or at compile time for Quarkus), the CDI extension scans the classpath. As soon as it finds an interface annotated with @RegisterAIService, it creates a synthetic bean. This bean is actually a LangChain4j proxy wrapped in a CDI wrapper. When you do @Inject, CDI gives you this proxy, and when you call a method, LangChain4j does the rest: LLM call, memory management, RAG, etc. Important point: the core of langchain4j-cdi does NOT depend on MicroProfile Config. It uses an SPI -- LLMConfig -- which is discovered via ServiceLoader. This SPI provides the configuration keys and values. MicroProfile Config is just ONE possible implementation, provided by the langchain4j-cdi-config module.
Emmanuel

WildFly alternative: The WildFly AI subsystem also allows configuration via standalone.xml

All configuration is externalized in MicroProfile Config properties. The pattern is simple: dev.langchain4j.cdi.plugin, the bean name, .class for the type, then .config.property for the builder parameters. The powerful thing is the lookup: you can reference other CDI beans directly from the config. lookup:@default for the default bean, lookup:@all for all, or lookup:name for a named bean. Zero code. It's 2026, not 2006. NOTE: For WildFly, there is also the WildFly AI subsystem that allows configuring models and MCP directly in standalone.xml. The AI Service interface


package com.example.demo1;

import dev.langchain4j.service.SystemMessage;
import dev.langchain4j.service.UserMessage;
import dev.langchain4j.cdi.RegisterAIService;

@RegisterAIService(chatModelName = "my-model")
public interface ChatAssistant {

    @SystemMessage("""
        You are a dwarf comedian in a tavern. You tell dwarf jokes
        full of self-deprecating humor about your height,
        your beard, your love of beer and gold, and your rivalries
        with elves. Your jokes are short and funny.
        """)
    String chat(@UserMessage String userMessage);
}
BACKUP if the demo doesn't work. Here is the interface. It's a contract: we define what the AI does, not how. The "how" is handled by CDI and LangChain4j. @SystemMessage defines the AI's behavior -- here a dwarf comedian who tells jokes. @UserMessage marks the user's parameter. @RegisterAIService tells CDI to create a bean for this interface. It's intentionally light and humorous to show that you can create ANY type of AI agent with this pattern.

The REST endpoint


@Path("/chat")
@ApplicationScoped
public class ChatResource {

    @Inject
    ChatAssistant assistant;  // ← CDI injection, classic!

    @POST
    @Consumes(MediaType.TEXT_PLAIN)
    @Produces(MediaType.TEXT_PLAIN)
    public String chat(String message) {
        return assistant.chat(message);
    }
}

No LangChain4j dependency in your business code!

And here is the REST endpoint. @Inject ChatAssistant, that's it. Note that the business code does NOT depend on LangChain4j. The import is just your interface. If tomorrow you switch AI frameworks, you change the implementation, not the business code. This is the principle of inversion of control taken to the extreme.

Streaming: real-time responses


@RegisterAIService(streamingChatModelName = "my-streaming-model")
public interface ChatAssistantStreaming {

    @SystemMessage("You are a creative assistant...")
    TokenStream chatStream(@UserMessage String userMessage);
}
    

// In the JAX-RS endpoint with Server-Sent Events
@GET
@Path("/stream")
@Produces(MediaType.SERVER_SENT_EVENTS)
public void chatStream(@QueryParam("message") String message,
                       @Context SseEventSink sink, @Context Sse sse) {
    TokenStream tokenStream = streamingAssistant.chatStream(message);

    tokenStream
        .onPartialResponse(token -> sink.send(sse.newEvent("token", token)))
        .onCompleteResponse(resp -> { sink.send(sse.newEvent("done", ""));
                                       sink.close(); })
        .onError(error -> { sink.send(sse.newEvent("error", error.getMessage()));
                            sink.close(); })
        .start();
}

ChatGPT effect: tokens arrive as they come! 🎬

Streaming allows displaying responses token by token, like in ChatGPT. The difference from the classic mode: we use streamingChatModelName instead of chatModelName, and the method returns a TokenStream instead of a String. The TokenStream integrates perfectly with JAX-RS Server-Sent Events. Each token is sent to the browser as soon as it arrives from the LLM. The user experience is much better: the user sees the response being built in real time instead of waiting 10 seconds to receive everything at once. DEMO: show stream.html with a dwarf song displaying progressively.

Vision: image analysis


@WebServlet("/uploadServlet")
@MultipartConfig
public class ImageAnalyzerServlet extends HttpServlet {

    @Inject
    @Named("vision-model")
    ChatModel visionModel;  // ← A simple ChatModel!

    @Override
    protected void doPost(HttpServletRequest request,
                         HttpServletResponse response) throws ... {
        Part file = request.getPart("file");

        UserMessage userMessage = UserMessage.from(
            TextContent.from("Describe this image in detail"),
            ImageContent.from(encodeBase64(file.getInputStream()),
                            file.getContentType())
        );

        ChatResponse answer = visionModel.chat(userMessage);
        response.getWriter().write(answer.aiMessage().text());
    }
}

Multimodal = text + image in the same message

Multimodal AI is the ability to analyze images in addition to text. With LangChain4j-CDI, it's incredibly simple: we inject a ChatModel configured to use a vision model (like llama3.2-vision or GPT-4 Vision). The important part: UserMessage.from() can take multiple contents -- TextContent for text, ImageContent for the image. The image must be base64-encoded. The vision model analyzes the image and responds in text. It works for describing images, extracting text (OCR), identifying objects, analyzing charts... Configuration: in microprofile-config.properties, we declare the vision-model with the appropriate vision model class. DEMO: show image.html, upload a photo, see the AI-generated description.

Tools : Function Calling


@ApplicationScoped
public class BookingTools {

    @Inject
    BookingService bookingService;

    @Tool("Search for available slots for a given date")
    public List<Slot> findAvailableSlots(
            @P("Date in yyyy-MM-dd format") String date) {
        return bookingService.getSlots(LocalDate.parse(date));
    }

    @Tool("Make a reservation")
    public Booking makeReservation(
            @P("Customer name") String name,
            @P("Desired date") String date) {
        return bookingService.book(name, LocalDate.parse(date));
    }
}

Tools are CDI beans → dependency injection in the tools! 🎉

Tools are the killer feature. They allow the AI to call your business code. And since Tools are CDI beans, they can themselves inject other services! BookingService, UserRepository, EmailSender... everything in your CDI container. This is where langchain4j-cdi shines compared to the manual approach: your tools are managed, transactional, observable.

MicroProfile Fault Tolerance


@RegisterAIService(chatModelName = "my-model")
public interface ResilientAssistant {

    @Retry(maxRetries = 3, delay = 1000)
    @Timeout(value = 30, unit = ChronoUnit.SECONDS)
    @Fallback(fallbackMethod = "fallbackChat")
    @CircuitBreaker(requestVolumeThreshold = 5,
                    failureRatio = 0.5)
    String chat(@UserMessage String message);

    default String fallbackChat(String message) {
        return "Service temporarily unavailable. "
             + "Please try again in a moment.";
    }
}
Emmanuel here. Look at the beauty of CDI: MicroProfile Fault Tolerance annotations work directly on your AI Services. No specific code, no adapter. @Retry: if the LLM times out or returns an error, we retry 3 times. @Timeout: no more than 30 seconds of waiting. @CircuitBreaker: after 5 calls with 50% failures, the circuit is cut. @Fallback: a fallback message if everything fails. All of this already exists in MicroProfile. We didn't invent anything. We just... injected it.
Emmanuel

Configuration in two parts. First part: MicroProfile Telemetry. We configure the OTLP endpoint (port 4318 HTTP), the service name, and the exporters for traces and metrics. Zero plumbing code -- everything goes through properties 🌍

MCP -- Model Context Protocol -- is the open standard for connecting AI agents to external tools. Think of it as JDBC but for AI. The important thing here is the toolProviderName in @RegisterAIService. It's what links the agent to the McpToolProvider. Configuration: two possible approaches 1. Via MicroProfile Config (properties) -- example shown here 2. Via the WildFly AI subsystem (standalone.xml) -- used in demo 3 Two main transports: stdio to launch a local process (like an MCP jar), or HTTP streamable/SSE for a remote server (that's what we use in our demo 3). Once again, zero plumbing code: you declare the MCP server (properties or XML), CDI injects the McpToolProvider, and the agent automatically discovers the tools exposed by the server. This opens the door to the entire ecosystem of MCP tools: databases, APIs, filesystem, web browsers...

WildFly AI Subsystem


<!-- standalone.xml - WildFly AI subsystem configuration -->
<subsystem xmlns="urn:jboss:domain:ai:1.0">
    <!-- Ollama model configuration -->
    <chat-language-models>
        <ollama-chat-model name="ollama"
            base-url="http://127.0.0.1:11434"
            model-name="llama3.1:8b"
            temperature="0.9"
            connect-timeout="600000"
            log-requests="true"
            log-responses="true"/>
    </chat-language-models>

    <!-- MCP SSE client configuration -->
    <mcp>
        <mcp-tool-provider name="mcp" mcp-clients="mcp-sse"/>
        <mcp-client-sse name="mcp-sse"
            socket-binding="mcp-sse"
            sse-path="/sse"
            connect-timeout="6000000"
            log-requests="true"
            log-responses="true"/>
    </mcp>
</subsystem>

<!-- Socket binding for the MCP server -->
<outbound-socket-binding name="mcp-sse">
    <remote-destination host="localhost" port="8090"/>
</outbound-socket-binding>

Native WildFly configuration -- Zero properties! 🚀

Emmanuel explains the WildFly AI subsystem. The WildFly AI Feature Pack adds a new "ai" subsystem to WildFly. It's a native extension that allows configuring: 1. Chat models (chat-language-models): - ollama-chat-model: connection to Ollama with base-url, model-name, temperature - Also supports OpenAI and other providers 2. MCP clients: - mcp-tool-provider: exposes MCP tools to CDI agents - mcp-client-sse: MCP client using Server-Sent Events (HTTP SSE) - socket-binding: points to the remote MCP server (localhost:8090) Everything is configured in standalone.xml. No need for microprofile-config.properties, no Java code for config. CDI agents simply reference the names: chatModelName="ollama" and toolProviderName="mcp". This is a real Jakarta EE/WildFly integration, not just a wrapper around LangChain4j.

Flow: Rolling dice via MCP SSE

1. Player: "Roll the dice"
2. JAX-RS endpoint → DwarfGameMaster.play()
3. LLM (Ollama) via WildFly AI subsystem → decides to use roll_multiple(count=3, sides=6)
4. McpToolProvider → JSON-RPC → MCP Dice Server (HTTP SSE on localhost:8090)
5. MCP Server rolls 3d6 → returns {"rolls": [4, 2, 1], "total": 7} via SSE
6. LLM analyzes the result → identifies "421" → responds to the player
7. Gunther: "BY MY BEARD! A perfect 421! 🎲🎉"

The MCP server is accessible via socket-binding configured in standalone.xml 🔧

Here is the complete flow of a dice roll in demo 3. The user clicks "Roll the dice" in the web interface. The JAX-RS endpoint injects DwarfGameMaster (the AI agent) and calls its play() method. The agent uses the "ollama" chat model configured in the WildFly AI subsystem. The LLM understands it needs to roll 3 six-sided dice, so it decides to call the roll_multiple tool with parameters count=3 and sides=6. The McpToolProvider (configured in the AI subsystem) intercepts this tool call and translates it into a JSON-RPC 2.0 call to the MCP server. The connection uses HTTP SSE (Server-Sent Events) via the "mcp-sse" socket-binding that points to localhost:8090. The mcp-client-sse establishes the SSE connection with the /sse path. The MCP server actually rolls 3 dice (random), calculates the total, and returns the result in JSON format via the SSE stream. The McpToolProvider returns the result to the LLM. The LLM analyzes the dice [4, 2, 1], identifies the "421" combination (the best in the game!), and generates a humorous response in the style of Gunther the dwarf tavern keeper. The response flows back up to the browser, and Gunther announces: "BY MY BEARD! A perfect 421!" The key point: everything is configured in standalone.xml via the WildFly AI subsystem. This is a real native WildFly integration, not just external properties.
